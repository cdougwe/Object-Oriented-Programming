import java.util.LinkedList;

class Logger {
    public void log(String msg) {
        System.out.println("Logging " + msg);
    }
}

class LoggerPool {
    private LinkedList pool = new LinkedList<>();
    private int maxSize;

    public LoggerPool(int maxSize) {
        this.maxSize = maxSize;
        for (int i = 0; i < maxSize; i++) {
            pool.add(new Logger());
        }
    }

    public synchronized Logger getLogger() {
        while (pool.isEmpty()) {
            try {
                wait();
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
        return pool.removeFirst();
    }

    public synchronized void releaseLogger(Logger logger) {
        pool.addLast(logger);
        notifyAll();
    }
}

class LoggingService {
    private LoggerPool pool;

    public LoggingService(int size) {
        pool = new LoggerPool(size);
    }
    public void logMessage(String message) {
        Logger logger = pool.getLogger();
        try {
            logger.log(message);
        } finally {
            pool.releaseLogger(logger);
        }
    }
}

public class Main {
    public static void main(String[] args) {
        LoggingService ls = new LoggingService(100);
        ls.logMessage("Hello");
        ls.logMessage("I am Tom");
    }
}

新設計有何優點?

這個設計是將「物件池（Object Pool）」與「多執行緒同步控制」結合，用來管理 Logger 這類可重複使用的資源。整體目的不是單純節省建立成本，而是讓多個執行緒在競爭資源時仍能安全、有序地運作。

在這個實作中，LoggerPool 一開始就建立固定數量的 Logger（例如 100 個），並放入 pool 中。當有需求時，透過 getLogger() 取得一個可用的 Logger；使用完畢後，透過 releaseLogger() 歸還。這樣的好處是避免在高頻率呼叫 log 時不斷建立新物件，同時也確保 Logger 的使用數量受到控制。

與前一個簡單版本相比，這裡的關鍵差異在於「同步機制」。getLogger() 與 releaseLogger() 都使用 synchronized，確保同一時間只有一個執行緒可以修改 pool。當 pool 為空時，getLogger() 不會直接回傳 null，而是呼叫 wait() 讓執行緒進入等待狀態，直到有其他執行緒釋放 Logger 並呼叫 notifyAll() 才會繼續執行。這種設計避免了忙等（busy waiting），也確保資源能被正確分配。

舉一個簡單情境：假設 pool 大小為 2，但同時有 3 個執行緒呼叫 logMessage()。前兩個可以正常取得 Logger，第三個會進入等待狀態，而不是失敗或產生錯誤。當其中一個執行緒完成並釋放 Logger，第三個才會被喚醒並繼續執行。這讓系統在高併發下仍能穩定運作。

另外，LoggingService 將 pool 封裝起來，外部只需要呼叫 logMessage()，不需要關心 Logger 的取得與釋放細節，降低使用複雜度。try-finally 的使用則確保即使中途發生例外，Logger 仍一定會被歸還，避免資源外洩。

整體來看，這個設計的優點在於同時達成三件事：限制資源數量、防止多執行緒衝突，以及提升系統穩定性。特別適用於「資源有限且需要被多執行緒共享」的場景，例如日誌系統、資料庫連線或執行緒池。!@#$

新設計有何優點?
這份程式主要是在做一個 Logger 的重複使用機制。一般如果每次要印 log 都 new 一個 Logger，其實在少量使用時沒什麼差，但如果是多執行緒或大量輸出時，會一直建立物件，多少會有負擔。所以這裡先建立固定數量的 Logger，之後需要用就從 pool 裡拿，用完再放回去。

比較特別的是這個版本有考慮到多執行緒的情況。像 getLogger() 有加 synchronized，代表同一時間只能有一個執行緒進來拿 Logger。如果 pool 已經被拿光了，程式也不會直接回傳 null，而是用 wait() 讓執行緒先停住，等有人把 Logger 還回來（releaseLogger() 裡的 notifyAll()），才繼續執行。這樣就不會發生搶資源或拿到空值的問題。

舉個簡單情況，如果 pool 裡只有 2 個 Logger，但同時有 3 個執行緒要寫 log，前兩個可以正常拿到，第三個就會先等著，不會出錯。等其中一個用完放回去，第三個才繼續跑。這種行為在實際系統裡其實滿常見的。

另外在 logMessage() 裡面用 try-finally，也是一個保險。就算中間發生例外，最後還是會把 Logger 放回 pool，不然久了 pool 會被用光。

整體來看，這個設計主要是在控制資源數量，順便處理多執行緒同時使用的問題。寫法沒有特別複雜，但把幾個常見問題（像資源耗盡、競爭）都一起處理掉。!@#$
