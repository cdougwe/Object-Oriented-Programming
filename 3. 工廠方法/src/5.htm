class GeneralMusicPlayer {
    protected IPlayerAdapter iAdapter;

    public GeneralMusicPlayer(IPlayerAdapter iPlayerAdapter) {
        this.iAdapter = iPlayerAdapter;
    }

    public final void start(String filename) {
        iAdapter.open(filename);
        iAdapter.play();
        try {
            Thread.sleep(1000);
        } catch (Exception e) {
        }
        iAdapter.stop();
        iAdapter.close();
    }
}

class MusicPlayerFactory {
    public GeneralMusicPlayer getPlayerType(String type) {
        if (type == null) {
            throw new IllegalArgumentException("PlayerType can not be a null value");
        }
        try {
            PlayerType playerType = PlayerType.valueOf(type.toUpperCase());
            IPlayerAdapter iPlayerAdapter = switch (playerType) {
                case MP3 -> new AdapterForMP3();
                case MIDI -> new AdapterForMidi();
                case WAV -> new AdapterForWAV();
            };
            return new GeneralMusicPlayer(iPlayerAdapter);
        } catch (Exception e) {
            throw new IllegalArgumentException("Unsuported type : " + type);
        }
    }
}

這個程式把「播放流程」、「播放格式」還有「物件建立」三件事分開處理，實際上是把 Adapter + Template Method + 工廠模式一起用在同一個案例裡。

先看核心流程，`GeneralMusicPlayer` 裡的 `start()` 把整個播放步驟固定住：開檔、播放、暫停一下、停止、關閉。這段流程不會因為檔案格式不同而改變，所以直接寫死在這裡，達到流程統一的效果。外面使用的人只要呼叫 `start()`，不用管細節。

再來是不同音樂格式的問題。MP3、Midi、WAV 原本各自有自己的方法名稱（像 `playMP3()`、`playMidi()`），完全不一致。這裡用 Adapter 把它們包成同一個接口 `IPlayerAdapter`，讓外部都用 `open / play / stop / close` 這組方法。這樣一來，`GeneralMusicPlayer` 就可以完全不用知道底層是 MP3 還是 WAV，直接呼叫統一接口就好。

整體設計的重點其實很明顯：播放流程被固定在一個地方（Template Method），不同格式被包成統一接口（Adapter），建立物件集中在工廠（Factory）。這樣做的結果是，之後如果要新增 FLAC 或其他格式，只需要寫一個新的 Adapter，然後在工廠補一行對應，原本的播放流程完全不用動。

簡單講，這個設計把「會變的東西」跟「不會變的東西」切開來處理。不會變的是播放流程，會變的是音樂格式。這樣分開之後，程式就變得比較穩定，也比較好擴充。

新設計有何優點?

這份程式我在做的時候，其實是想解決一個很直觀的問題：不同音樂格式（MP3、Midi、WAV）的方法名稱都不一樣，如果直接在主程式裡操作會變得很亂，而且每多一種格式就要多寫一堆判斷。

所以我先做了一件事，把所有播放器統一成一個接口 `IPlayerAdapter`，讓外面都只用 `open、play、stop、close` 這幾個方法。像 MP3 原本是 `playMP3()`，我就用 Adapter 幫它轉成 `play()`，其他格式也是一樣的概念。這樣主程式就不用管底層到底是哪一種格式，寫起來會乾淨很多。

接著我把播放流程固定在 `GeneralMusicPlayer` 裡面。因為不管是 MP3 還是 WAV，其實流程都一樣：開檔 → 播放 → 等一下 → 停止 → 關閉。所以我直接把這一整段寫在 `start()` 裡面。這樣好處是之後就不用每個地方都重寫一次流程，也比較不會漏掉某個步驟。

然後物件建立的部分我用了一個工廠 `MusicPlayerFactory` 來處理。主程式只要丟一個字串進去，就可以拿到對應的播放器，不用自己去 new Adapter 或組裝物件。再加上 enum 去限制類型，像 "mp3"、"MP3" 都可以處理，也比較不容易出錯。

實際跑的時候可以看到，不管是 WAV、MP3 還是 Midi，主程式寫法完全一樣，只是換一個 type 而已，其他都不用改。這也是我覺得這個設計比較有感的地方。

整體來說，這個設計把「播放流程」、「格式差異」、「物件建立」分開處理。未來如果要加新格式，其實只要多寫一個 Adapter，然後在工廠補上去就好，原本的程式幾乎不用動。這樣在改功能或擴充的時候會比較輕鬆，也比較不容易改壞其他地方。
