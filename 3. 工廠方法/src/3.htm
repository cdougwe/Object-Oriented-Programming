新設計有何優點?

這個程式示範了如何使用工廠模式搭配策略模式，實作文件匯出的功能，讓程式能夠依照不同檔案類型靈活輸出。

我先定義一個 `Exporter` 介面，裡面有 `writeToFile(String data, String filename)` 方法，代表將資料寫入檔案。接著建立兩個實作類別 `PDFExporter` 與 `CSVExporter`，分別負責 PDF 與 CSV 格式的輸出，輸出時會顯示對應訊息，例如「寫入 PDF檔案: Hello 到 file1.pdf」。

`Document` 類別持有一個 `Exporter`，並提供 `export` 方法，呼叫內部的 `Exporter` 完成資料寫入。這樣可以把匯出流程固定下來，而不同的檔案格式則由 `Exporter` 實作細節決定，符合策略模式的概念。

`DocumentFactory` 是工廠類別，透過傳入的字串判斷檔案類型，回傳對應的 `Document`。如果傳入 `"PDF"` 就建立 PDF 文件匯出服務，傳入 `"CSV"` 則建立 CSV 文件匯出服務。這樣做的好處是，程式主體不用關心細節，只要告訴工廠要哪種檔案，就能自動取得對應服務，未來若增加新格式，只需新增 `Exporter` 實作即可。

在 `main` 方法中，我先用工廠建立 PDF 文件匯出，呼叫 `export("Hello", "file1.pdf")`，顯示「寫入 PDF檔案: Hello 到 file1.pdf」。接著建立 CSV 文件匯出，呼叫 `export("Hello", "file2.csv")`，顯示「寫入 CSV: Hello 到 file2.csv」。整體設計把匯出流程與格式細節分離，提高了程式的可擴充性與維護性。

enum FileType {
    PDF, CSV;
}

interface Exporter {
    void writeToFile(String data, String filename);
}

class PDFExporter implements Exporter {
    @Override
    public void writeToFile(String data, String filename) {
        System.out.println("寫入 PDF檔案: " + data + " 到 " + filename);
    }
}

class CSVExporter implements Exporter {
    @Override
    public void writeToFile(String data, String filename) {
        System.out.println("寫入 CSV: " + data + " 到 " + filename);
    }
}

abstract class Document {
    private Exporter exporter;

    final void export(String data, String filename) {
        exporter = genExporter(); // 生成匯出工具
        exporter.writeToFile(data, filename); // 執行匯出
    }

    abstract Exporter genExporter();
}

class PDF extends Document {
    @Override
    Exporter genExporter() {
        return new PDFExporter();
    }
}

class CSV extends Document {
    @Override
    Exporter genExporter() {
        return new CSVExporter();
    }
}

class DocumentFactory {
    public Document getType(String fileType) {
        if (fileType == null)
            throw new IllegalArgumentException("FillType should not be a null value");
        try {
            FileType type = FileType.valueOf(fileType.toUpperCase());
            return switch (type) {
                case PDF -> new PDF();
                case CSV -> new CSV();
            };
        } catch (Exception e) {
            throw new IllegalArgumentException("Unsupported file type:" + fileType);
        }
    }
}

public class Main {
    public static void main(String[] args) {
        DocumentFactory documentFactory = new DocumentFactory();
        Document exporter = documentFactory.getType("Pdf");
        exporter.export("Hello", "file1.pdf");
        exporter = documentFactory.getType("CSV");
        exporter.export("Hello", "file2.csv");
    }
}

新設計有何優點?

這個新設計最大的優點在於彈性大、維護容易。以前如果每種檔案格式都寫死在程式裡，程式碼會變得很雜，而且流程幾乎一樣，但如果想改或新增格式，就得修改很多地方，非常容易出錯。現在用抽象類別統一流程，PDF、CSV 或將來新增的格式，只要自己實作匯出的細節就行，主程式根本不用改，這樣程式結構就清楚很多，也方便測試。

另外，這種設計模組化明顯。各種格式互不干擾，改 PDF 邏輯不會影響 CSV，新增其他格式也不用碰到舊程式碼。再加上工廠模式，主程式只要告訴工廠想要哪種檔案，它就幫你產生對應物件，主程式不需要知道底層細節。這樣不僅降低出錯率，也省下很多維護成本。

最後，這個設計還很具擴充性。如果以後有新的檔案格式，例如 Excel、TXT，只要新增對應的類別和工廠設定就好，主程式完全不用動，對應的流程也自然套用抽象類別的規範。整體來說，比傳統把所有流程寫死的方式穩定、乾淨，也方便長期維護，程式的可讀性和可擴充性都大幅提升。!@#$
